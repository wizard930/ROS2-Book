# 8장 DDS의 QoS

---
## 8.1. DDS의 서비스 품질(Qos, Quality of Service)
- DDS에서 “서비스 품질(QoS)”이 의미하는 바이다
- DDS QoS는 데이터 통신을 “그냥 보내기”가 아니라 요구조건에 맞게 전달 특성을 설계하는 기능이다.
- QoS를 통해 유실 허용/전달 보장, 지연/안정성 트레이드오프, 과거 데이터 제공 여부, 생존/끊김 감지 같은 통신 품질을 선택하는 구조이다.

---
## 8.2. QoS 종류(대표 항목)
![alt text](p106-22개QoSPolicy.png)

### 1) Reliability(신뢰성)
- **BEST_EFFORT**: 빠르지만 유실을 허용하는 방식이다.
- **RELIABLE**: 전달 보장을 우선하는 방식이다(재전송/버퍼로 지연이 늘 수 있는 전제이다).

### 2) History(히스토리)
- **KEEP_LAST**: 최신 N개만 유지하는 방식이다(Depth와 함께 사용되는 구조이다).
- **KEEP_ALL**: 가능한 한 모두 유지하는 방식이다(메모리 사용 증가 전제가 있다).

### 3) Depth(큐 깊이)
- **KEEP_LAST**에서 “몇 개를 유지할지”를 정하는 값이다.
- 예: Depth=10이면 최근 10개 데이터만 보관하는 구조이다.

### 4) Durability(지속성)
- VOLATILE: 현재 연결된 구독자에게만 전달되는 성격이다.
- TRANSIENT_LOCAL: 늦게 참여한 구독자에게도 최근 데이터를 제공할 수 있는 성격이다.

### 5) Deadline(기한)
- “이 데이터는 일정 주기 안에 도착해야 한다”는 조건을 정의하는 QoS이다.
- 기한 위반을 시스템이 감지할 수 있는 구조이다.

### 6) Liveliness(생존성)
- 발행자/구독자가 살아있는지 확인하고 끊김을 감지하는 QoS이다.

### 7) Lifespan(수명)
- 데이터가 유효한 시간(만료 시간)을 정의하는 QoS이다.

### 8) Latency Budget(지연 예산)
- 허용 가능한 지연 예산을 제시하는 QoS이다(구현체 최적화 힌트 성격이 될 수 있다).

### 9) Transport Priority(전송 우선순위)
- 데이터 전송 우선순위를 표현하는 QoS이다(지원 여부는 구현체/환경에 따라 다를 수 있다).

## 8.3. ROS 2 QoS “정책(Policy)” 종류
### 1) History / Depth
- **History**: 큐에 데이터를 어떻게 저장할지 정하는 정책이다.
  - `KEEP_LAST`: 최근 N개만 유지하는 정책이다(Depth로 N을 지정하는 구조이다).
  - `KEEP_ALL`: 가능한 한 모두 유지하는 정책이다.
- **Depth**: KEEP_LAST에서 유지할 개수(N)를 의미하는 값이다.
```cpp
rclcpp::QoS(rclcpp::KeepLast(10));
```
```py
qos_profile = QoSProfile(history=QoSHistoryPolicy.KEEP_LAST, depth=10)
```
### 2) Reliability
- **BEST_EFFORT**: 빠르지만 유실을 허용하는 정책이다.
- **RELIABLE**: 전달 보장을 우선하는 정책이다(재전송/버퍼로 지연이 늘 수 있는 전제이다).
```cpp
rclcpp::QoS(rclcpp::KeepAll).best_effort();
```
```py
qos_profile = QoSProfile(reliability=QoSReliabilityPolicy.BEST_EFFORT)
```
### 3) Durability
- **VOLATILE**: 현재 연결된 구독자에게만 전달되는 정책이다.
- **TRANSIENT_LOCAL**: 늦게 참여한 구독자에게도 최근 데이터를 제공할 수 있는 정책이다.

### 4) Deadline
- 데이터가 “정해진 시간 안에” 도착해야 한다는 시간 제약을 정의하는 정책이다.
- 기한 위반을 감지하는 데 사용되는 정책이다.
```cpp
rclcpp::QoS(rclcpp::KeepAll).transient_local();
```
```py
qos_profile = QoSProfile(durability=QoSDurabilityPolicy.TRANSIENT_LOCAL)
```
### 5) Lifespan
- 데이터가 유효한 시간(만료 시간)을 정의하는 정책이다.
- 오래된 데이터가 전달되는 것을 줄이는 데 사용되는 정책이다.
```cpp
rclcpp::QoS(10).reliable().transient_local().lifespan(10ms);
```
```py
qos_profile = QoSProfile(lifespan=Duration(0.01))
```
### 6) Liveliness (+ Lease Duration)
- 참여자(발행자/구독자)의 생존 여부를 감지하는 정책이다.
- Lease Duration은 “얼마 동안 신호가 없으면 끊긴 것으로 볼지”의 기준 시간 성격이다.
```cpp
rclcpp::QoS qos_profile(10);
qos_profile
  .liveliness(RMW_QOS_POLICY_LIVELINESS_AUTOMATIC)
  .liveliness_lease_duration(1000ms);
```
```py
qos_profile = QoSProfile(
  liveliness=AUTOMATIC,
  liveliness_lease_duration=Duration(1.0))
```

## 8.4. rmw_qos_profile 사용과 유저 QoS 프로파일
- ROS 2에서 QoS를 설정할 때 자주 등장하는 두 가지 방식인 **1) rmw_qos_profile(저수준 QoS)** 과 **2) 유저 QoS 프로파일(상위 API QoS)** 을 간단히 정리한 내용이다.
- ROS 2 QoS는 “통신을 설계하는 옵션”이며, 위 6가지는 **버퍼(History), 유실/보장(Reliability), 과거 데이터(Durability), 주기 감시(Deadline), 데이터 유효기간(Lifespan), 생존 감지(Liveliness)** 를 각각 담당하는 옵션이다.

### 1)rmw_qos_profile
| 항목 | Default | Sensor Data | Service | Action Status | Parameters | 
-Parameter Events |
|---|---|---|---|---|---|---|
| Reliability | RELIABLE | BEST_EFFORT | RELIABLE | RELIABLE | RELIABLE | RELIABLE |
| History | KEEP_LAST | KEEP_LAST | KEEP_LAST | KEEP_LAST | KEEP_LAST | KEEP_LAST |
| Depth (History Depth) | 10 | 5 | 10 | 1 | 1,000 | 1,000 |
| Durability | VOLATILE | VOLATILE | VOLATILE | TRANSIENT_LOCAL | VOLATILE | VOLATILE |
---
### Default
- 일반적인 토픽 통신에 쓰기 무난한 기본값이다.
- **RELIABLE + KEEP_LAST(10) + VOLATILE**로 “대체로 안정적으로 받고, 최근 10개만 버퍼링”하는 성향이다.

### Sensor Data
-
- 카메라/LiDAR/IMU처럼 고주기 센서 스트림에 맞춘 프리셋이다.
- **BEST_EFFORT + KEEP_LAST(5)**로 “지연을 줄이고, 필요하면 드롭을 허용하며, 최신 위주로 받는” 성향이다.

```c
static const rmw_qos_profile_t rmw_qos_profile_sensor_data =
{
  RMW_QOS_POLICY_HISTORY_KEEP_LAST,
  5,
  RMW_QOS_POLICY_RELIABILITY_BEST_EFFORT,
  RMW_QOS_POLICY_DURABILITY_VOLATILE,
  RMW_QOS_DEADLINE_DEFAULT,
  RMW_QOS_LIFESPAN_DEFAULT,
  RMW_QOS_POLICY_LIVELINESS_SYSTEM_DEFAULT,
  RMW_QOS_LIVELINESS_LEASE_DURATION_DEFAULT,
  false
};
```
- 코드에서 사용하기 위한 임포트
```py
from rclpy.qos import qos_profile_sensor_data
```
- qos_profile_sensor_data 매개변수 사용
- 각각 msg_type, topic_name, qos_profile 이다
```cpp
self.sensor_publisher=self.create_publisher(Int8MultiArray,'sensor', 
-qos_profile_sensor_data)
```
### Service
- request/response 성격의 서비스 통신에 맞춘 프리셋이다.
- **RELIABLE + KEEP_LAST(10)**로 “요청/응답 유실을 줄이는” 성향이다.
### Action Status
- 액션의 상태(Status) 공유에 맞춘 프리셋이다.
- **RELIABLE + KEEP_LAST(1) + TRANSIENT_LOCAL**로 “최신 상태 1개를 유지하고, 늦게 들어온 구독자도 최근 상태를 받게 하는” 성향이다.
### Parameters
- 파라미터 관련 토픽/통신에 맞춘 프리셋이다.
- **RELIABLE + KEEP_LAST(1000)**로 “손실을 줄이고, 이벤트/변경 히스토리를 많이 담을 수 있게 버퍼를 크게 잡는” 성향이다.
### Parameter Events
- 파라미터 변경 이벤트 스트림에 맞춘 프리셋이다.
- **RELIABLE + KEEP_LAST(1000)**로 “파라미터 이벤트 누락을 줄이고, 많은 이벤트를 버퍼링”하는 성향이다.

### 2)유저 QoS 프로파일
- **유저 QoS 프로파일(rclcpp/rclpy QoS)** 은 애플리케이션에서 쓰는 “사용자 친화 QoS”이고, 내부적으로 **rmw_qos_profile_t로 변환되어 DDS(또는 다른 RMW)로 전달**되는 구조이다.

- 유저 임포트
```py
from rclpy.qos import QoSDurabilityPolicy
from rclpy.qos import QoSHistoryPolicy
from rclpy.qos import QoSProfile
from rclpy.qos import QoSReliabilityPolicy
```
- 사용자측 커스텀
```py
QoS_RK10V = QoSProfile(
    reliability=QoSReliabilityPolicy.RELIABLE,
    history=QoSHistoryPolicy.KEEP_LAST,
    depth=10,
    durability=QoSDurabilityPolicy.VOLATILE
)
```
- 사용
```py
self.sensor_publisher = self.create_publisher(Int8MultiArray, 'sensor', QoS_RK10V)
```

## 8.5. QoS 테스트
- ROS 2에서 **QoS(Quality of Service)** 설정이 의도대로 동작하는지 확인하기 위한 **기본 테스트 방법**을 간단히 정리한 내용이다.
- QoS 테스트의 목적으로 토픽 통신이 “된다/안 된다”가 아니라 **유실·지연·버퍼링·과거 데이터 제공** 같은 품질이 요구사항대로 동작하는지 확인하는 목적이다.
- 퍼블리셔(Pub)와 서브스크라이버(Sub)의 QoS가 맞지 않아 **통신이 안 되거나(특히 Reliability)**, **메시지 드롭이 증가**하는 문제를 빠르게 찾는 목적이다.
