13장 ROS 2 액션 (Action) 학습 리포트

1.1 액션(Action)의 정의 및 개념

**액션(Action)**은 노드 A(액션 클라이언트/Action Client)와 노드 B(액션 서버/Action Server) 간의 비동기식 및 동기식 양방향 메시지 송수신 방식의 통신이다. 액션은 특정 목표를 지정하고, 그 목표를 달성하기 위해 상대적으로 긴 시간이 소요되는 로봇 태스크를 수행할 때 사용된다.

액션의 데이터 구조는 다음과 같은 세 가지 핵심 요소로 구성된다:

* 액션 목표(Goal): 클라이언트가 서버에 전달하는 작업 지시 데이터이다.
* 액션 피드백(Feedback): 서버가 작업을 수행하는 동안 클라이언트에게 실시간으로 전달하는 중간 결괏값이다.
* 액션 결과(Result): 작업이 최종적으로 완료되었을 때 서버가 클라이언트에게 전달하는 최종 결괏값이다.

1.2 액션의 복합 통신 구조

ROS 2 액션은 독립적인 단일 통신이 아니라, **토픽(Topic)**과 **서비스(Service)**의 기능을 정교하게 결합하여 구현된 복합 구조이다.

* 구현의 차이점: 과거 ROS 1에서는 액션의 모든 데이터(목표, 피드백, 결과)를 토픽으로만 주고받았다. 그러나 ROS 2에서는 비동기 통신 중 발생하는 타이밍 이슈를 해결하고 통신의 신뢰성을 높이기 위해 서비스 방식을 적극적으로 도입하였다.
* 내부 구성 요소:
  * Action Client: 3개의 **서비스 클라이언트(Service Client)**와 2개의 **토픽 서브스크라이버(Topic Subscriber)**로 구성된다.
  * Action Server: 3개의 **서비스 서버(Service Server)**와 2개의 **토픽 퍼블리셔(Topic Publisher)**로 구성된다.
* 상세 통신 매핑:
  * 서비스(Service): 목표 전달(send_goal), 목표 취소(cancel_goal), **결과 받기(get_result)**에 사용된다. (예: _action/send_goal은 RotateAbsolute_SendGoal 인터페이스를 사용)
  * 토픽(Topic): 피드백(feedback) 전송과 목표의 현재 상태(status) 확인에 사용된다. (예: _action/feedback은 RotateAbsolute_FeedbackMessage 인터페이스를 사용)

1.3 Goal State Machine (목표 상태 머신)

ROS 2 액션은 비동기와 동기 방식이 혼재된 복잡한 제어 프로세스를 추적하기 위해 목표 상태(goal_state) 개념을 도입하였다. **목표 상태 머신(Goal State Machine)**을 통해 액션 클라이언트는 서버의 작업 진행 단계를 명확히 파악할 수 있다.

* 상태 분류:
  * 활성 상태(Active State): 프로세스가 현재 진행 중인 상태이다.
    * ACCEPTED: 서버가 목표를 수락함 (서버 트리거/Server Triggered)
    * EXECUTING: 목표를 수행 중
    * CANCELING: 취소 요청을 받고 처리 중
  * 종료 상태(Terminal State): 프로세스가 완료되어 종료된 상태이다.
    * SUCCEEDED: 목표 달성 성공
    * ABORTED: 작업 중단
    * CANCELED: 취소 완료
* 트리거(Trigger): 상태 전환은 **클라이언트 트리거(Client Triggered)**인 send_goal, cancel_goal 요청과 서버 내부의 판단에 따른 **서버 트리거(Server Triggered)**에 의해 이루어진다.

1.4 액션 인터페이스 (Action Interface, .action)

액션 인터페이스는 메시지(msg)와 서비스(srv) 인터페이스의 확장형으로, .action 파일에 정의된다. 이 인터페이스는 두 개의 구분자(---)를 사용하여 세 가지 데이터 영역을 명확히 구분한다.

turtlesim/action/RotateAbsolute.action의 실제 데이터 구조 예시는 다음과 같다:

# 액션 목표(Goal)
float32 theta
---
# 액션 결과(Result)
float32 delta
---
# 액션 피드백(Feedback)
float32 remaining


위 구조에서 theta는 거북이가 회전할 절대 각도를 의미하며, delta는 최종 변위, remaining은 목표까지 남은 회전량을 실시간으로 보고하는 데 사용된다.

1.5 주요 CLI 명령어 및 실습 정리

실습은 turtlesim_node와 turtle_teleop_key를 활용하여 진행한다. 텔레오퍼레이션 노드에서 특정 키를 누르면 rotate_absolute 액션의 목푯값이 서버로 전달된다.

* 주요 키 매핑 정보:
  * G: 3시 방향 (theta: 0.0 rad), 액션의 기준 각도이다.
  * R: 12시 방향 (theta: 1.5708 rad).
  * F: 현재 진행 중인 액션 목표를 취소(cancel_goal)한다.
* 핵심 CLI 명령어:
  * ros2 node info: 노드가 보유한 액션 서버와 클라이언트의 이름 및 인터페이스 형태를 확인한다.
  * ros2 action list -t: 시스템 내 실행 중인 액션 목록과 타입을 표시한다.
  * ros2 action info <action_name>: 특정 액션의 서버/클라이언트 이름과 연결 개수를 확인한다.
  * ros2 action send_goal <action_name> <action_type> "<values>": 명령행에서 직접 목표를 전달한다. (예: ros2 action send_goal /turtle1/rotate_absolute turtlesim/action/RotateAbsolute "{theta: 1.5708}")
  * --feedback 옵션: send_goal 시 이 옵션을 추가하면 액션 **UID(Unique ID)**와 함께 남은 회전량 등의 피드백 데이터를 실시간으로 출력한다.

1.6 토픽, 서비스, 액션 비교

ROS 2의 세 가지 통신 방식을 상세히 비교하면 다음과 같다.

구분	토픽 (Topic)	서비스 (Service)	액션 (Action)
연속성	연속성	일회성	복합 (토픽 + 서비스)
방향성	단방향	양방향	양방향
동기성	비동기	동기	동기 + 비동기
다자간 연결	1:1, 1:N, N:1, N:N	1:1	1:1
노드 역할	퍼블리셔 / 서브스크라이버	서버 / 클라이언트	서버 / 클라이언트
동작 트리거	퍼블리셔 (Publisher)	클라이언트 (Client)	클라이언트 (Client)
인터페이스	msg 인터페이스	srv 인터페이스	action 인터페이스
사용 예	센서 데이터, 로봇 상태	모드 설정, 간단한 제어	목적지 이동, 복합 태스크

1.7 학습 요약

* 액션은 목표(Goal), 피드백(Feedback), **결과(Result)**를 포함하는 고도화된 양방향 통신 방식이다.
* ROS 2는 ROS 1과 달리 서비스와 토픽을 혼합하여 액션을 구현함으로써 통신의 타이밍과 신뢰성 문제를 해결하였다.
* **목표 상태 머신(Goal State Machine)**을 통해 복잡한 비동기 작업의 진행 단계를 체계적으로 관리하고 추적할 수 있다.
* 작업 시간이 길고 중간 과정 모니터링이 필수적인 자율 주행이나 물건 파지 작업 등에 가장 적합한 통신 메커니즘이다.
