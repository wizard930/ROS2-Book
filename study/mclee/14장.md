14장 ROS 2 인터페이스 정의 및 구조 요약 리포트

14.1 ROS 2 인터페이스 개요

14.1.1 핵심 정의 및 IDL의 도입

ROS 2 인터페이스(Interface)는 노드 간 데이터를 주고받기 위해 사용되는 정형화된 데이터의 형태를 의미합니다. 이는 토픽(Topic), 서비스(Service), 액션(Action) 통신에서 전송되는 데이터의 규격을 결정하는 핵심 요소입니다. ROS 2에서는 기존 ROS 1의 형식을 계승하면서도, **IDL(Interface Definition Language)**을 새롭게 도입하였습니다. IDL은 특정 프로그래밍 언어에 종속되지 않는 독립적인 인터페이스 정의를 가능하게 하며, 이를 통해 정의된 데이터 구조는 빌드 과정을 거쳐 C++, Python 자료형 및 DDS(Data Distribution Service) 자료형으로 자동 매핑됩니다.

14.1.2 자료형 구조의 유연성

인터페이스는 다음과 같은 계층적 구조를 가질 수 있어 복잡한 로봇 데이터를 효율적으로 표현합니다.

* 단순 자료형: 정수, 부동 소수점, 불리언 등 가장 기초적인 데이터 단위.
* 메시지 중첩 구조: 특정 메시지 내에 이미 정의된 다른 메시지를 포함하는 구조 (예: Twist 메시지 내의 Vector3).
* 배열 구조: 동일한 자료형을 나열하는 방식이며, 고정 크기(Static)와 가변 크기(Dynamic) 배열을 모두 지원합니다.

14.1.3 언어별 자료형 매핑 정보

14.2 메시지 인터페이스 (msg)

14.2.1 용도 및 특징

메시지 인터페이스는 주로 토픽 통신에서 데이터를 비동기적으로 스트리밍할 때 사용됩니다. 파일 확장자는 .msg이며, ROS 2의 가장 기본이 되는 데이터 단위입니다.

14.2.2 중첩 구조 분석: geometry_msgs/msg/Twist

로봇의 속도 제어에 사용되는 Twist 메시지는 중첩 구조의 대표적인 사례입니다.

* 구조: Vector3 linear와 Vector3 angular라는 두 개의 하위 메시지로 구성됩니다.
* 세부 데이터: Vector3는 내부적으로 float64 x, y, z 필드를 가집니다.
* 결과: 최종적으로 6개의 float64 값이 조합되어 로봇의 병진 및 회전 속도를 정의합니다.

14.2.3 인터페이스 관련 CLI 명령어

* ros2 interface show: 특정 인터페이스의 내부 필드와 데이터 타입을 상세히 확인합니다.
* ros2 interface list: 시스템 내 가용한 모든 msg, srv, action 인터페이스를 나열합니다.
* ros2 interface package: 지정된 특정 패키지에 포함된 인터페이스 목록을 확인합니다.
* ros2 interface packages: 인터페이스를 하나라도 포함하고 있는 패키지들의 목록을 출력합니다.
* ros2 interface proto: 특정 인터페이스의 기본 템플릿(Prototype) 형태를 출력하여 메시지 작성을 돕습니다.

14.3 서비스 인터페이스 (srv)

14.3.1 구조적 특징

서비스 인터페이스는 요청(Request)과 응답(Response)이라는 독립된 두 데이터 구조를 포함합니다. 확장자는 .srv이며, 메시지 인터페이스의 확장된 형태입니다.

14.3.2 데이터 구분 메커니즘

파일 내에서 서비스 요청 데이터와 응답 데이터를 물리적으로 나누기 위해 하이픈 세 개로 이루어진 **구분자(---)**를 사용합니다. 서비스 클라이언트는 구분자 윗부분의 데이터를 전송하고, 서비스 서버는 작업을 수행한 후 구분자 아랫부분의 데이터를 회신합니다.

14.3.3 실습 사례: turtlesim/srv/Spawn.srv

거북이를 특정 위치에 생성하는 서비스의 인터페이스 구조는 다음과 같습니다.

float32 x
float32 y
float32 theta
string name
---
string name

14.4 액션 인터페이스 (action)

14.4.1 통신 메커니즘 및 3단계 구조

액션은 장시간 소요되는 태스크를 관리하기 위해 토픽과 서비스의 장점을 결합한 인터페이스입니다. 목표(Goal), 결과(Result), 피드백(Feedback)을 정의하기 위해 두 개의 구분자(---)를 사용합니다.

14.4.2 내부 구현 분석

아키텍처 관점에서 액션은 단순한 단일 통신이 아니라, 3개의 서비스와 2개의 토픽이 결합된 복합체입니다.

* 서비스(Service) 영역: send_goal(목표 전달), cancel_goal(목표 취소), get_result(결과 요청)
* 토픽(Topic) 영역: feedback(중간 데이터 전송), status(액션 상탯값 전송)

이러한 구조는 **Goal State Machine(목표 상태 머신)**을 구동하게 하여, 클라이언트가 액션의 진행 프로세스를 정교하게 추적하고 제어할 수 있도록 돕습니다.

14.4.3 실습 사례: turtlesim/action/RotateAbsolute.action

거북이를 절대 각도로 회전시키는 액션의 인터페이스 구조입니다.

float32 theta
---
float32 delta
---
float32 remaining

14.5.2 핵심 요약

결론적으로 ROS 2의 모든 인터페이스는 가장 기초적인 .msg 인터페이스의 확장 및 변형된 형태입니다. 도입된 IDL 체계는 개발자로 하여금 언어에 구애받지 않고 일관된 방식으로 데이터를 정의하게 하며, 이는 복잡한 멀티 로봇 시스템 및 산업용 애플리케이션에서 높은 유연성과 안정성을 제공하는 기반이 됩니다.
