15장. ROS 2 토픽/서비스/액션 정리 및 비교

1. 통신 방식 개요 및 분석 목적

ROS 2 로봇 시스템 아키텍처에서 노드 간의 유기적인 데이터 교환은 전체 시스템의 신뢰성과 실시간성을 결정짓는 핵심 설계 요소입니다. ROS 2는 다양한 로봇 제어 요구사항을 충족하기 위해 **토픽(Topic), 서비스(Service), 액션(Action)**이라는 세 가지 주요 통신 메커니즘을 제공합니다.

본 리포트의 목적은 각 통신 방식의 기술적 특성을 명확히 대조하고, 시스템 설계 시 발생할 수 있는 스레드 차단(Blocking)이나 리소스 낭비를 방지하기 위한 최적의 통신 방식 선택 기준을 제시하는 데 있습니다. 이를 분석하기 위해 다음 7가지 핵심 지표를 정의합니다.

* 연속성: 데이터 전송이 스트리밍 형태인지(연속), 단일 이벤트성인지(일회성) 정의
* 방향성: 데이터의 흐름이 한 방향인지, 요청-응답의 양방향 구조인지 확인
* 동기성: 송신 측이 응답을 기다리며 대기하는지(동기), 대기 없이 즉시 다음 로직을 수행하는지(비동기) 구분
* 다자간 연결: 노드 간의 연결 형태(1:N, 1:1 등) 분석
* 노드 역할: 통신 주체(Publisher/Subscriber, Server/Client)의 정의
* 동작 트리거: 통신을 능동적으로 시작하는 주체 확인
* 인터페이스: 통신 데이터 구조를 정의하는 파일 형식 및 확장자

2. 토픽, 서비스, 액션 상세 비교

2.1 종합 비교표

소스 자료 [표 15-1]을 바탕으로 정리한 세 가지 통신 방식의 기술적 사양은 다음과 같습니다.

구분 지표	토픽 (Topic)	서비스 (Service)	액션 (Action)
연속성	연속	일회성	복합 (토픽 + 서비스)
방향성	단방향	양방향	양방향
동기성	비동기	동기	동기 + 비동기 (혼합)
다자간 연결	1:1, 1:N, N:1, N:N	1:1	1:1
노드 역할	퍼블리셔, 서브스크라이버	서버, 클라이언트	서버, 클라이언트
동작 트리거	퍼블리셔	클라이언트	클라이언트
인터페이스	msg 인터페이스	srv 인터페이스	action 인터페이스
CLI 명령어	ros2 topic	ros2 service	ros2 action

2.2 설계 관점의 주요 차이점 분석

* 연속성 및 방향성: 토픽은 센서 데이터와 같이 지속적으로 발생하는 정보를 단방향 스트림으로 전송하는 데 최적화되어 있습니다. 반면 서비스와 액션은 특정 요청(Request)에 따른 결과(Response/Result)를 보장해야 하므로 양방향 구조를 채택합니다.
* 동기성(Synchronicity)의 기술적 의미: 서비스는 기본적으로 동기 방식을 사용합니다. 이는 클라이언트가 요청 후 응답을 받을 때까지 대기함을 의미하며, 설계 시 실시간 루프 내에서 서비스 호출이 스레드 락(Thread-locking)을 유발하지 않도록 주의해야 합니다. 반면 액션은 동기+비동기 혼합 방식을 채택하여, 목표 전달과 결과 수신은 신뢰성 있는 서비스로 처리하고 과정 피드백은 비차단(Non-blocking) 토픽으로 처리함으로써 시스템 효율성을 극대화합니다.
* 다자간 연결 구조: 토픽은 발행-구독 모델을 통해 다수의 노드가 동일한 데이터를 공유하는 유연한 확장이 가능하지만, 서비스와 액션은 명확한 요청 주체와 수행 주체 간의 1:1 대응 관계를 기반으로 동작합니다.

3. 통신 방식별 인터페이스 및 데이터 구조

3.1 인터페이스 비교 및 실무 사례

모든 인터페이스는 기본 msg 구조의 확장형이며, 시스템에서 사용되는 구체적인 예시는 다음과 같습니다.

항목	msg (토픽)	srv (서비스)	action (액션)
확장자	*.msg	*.srv	*.action
주요 데이터	data	request / response	goal / result / feedback
실무 예시	geometry_msgs/msg/Twist	turtlesim/srv/Spawn	turtlesim/action/RotateAbsolute

3.2 데이터 구분 방식 및 필드 정의

각 인터페이스 파일은 fieldtype fieldname 형식을 따르며, 용도에 따라 하이픈(---) 구분자를 사용하여 논리적 영역을 분리합니다.

* 서비스(srv): 한 개의 구분자(---)를 사용하여 **요청(Request)**과 응답(Response) 영역을 나눕니다.
* 액션(action): 두 개의 구분자(---)를 사용하여 목표(Goal), 결과(Result), **피드백(Feedback)**의 세 가지 영역을 정의합니다.
* 공통 특성: 인터페이스 내부에 다른 메시지 타입을 포함하거나 배열(Array) 구조를 정의할 수 있어, 복잡한 로봇 데이터 구조 설계가 가능합니다.

4. 실무 활용 사례 및 선택 가이드

4.1 통신 방식별 최적 사용 예시

* 토픽 (Topic): 실시간 데이터 스트리밍
  * 라이다/카메라 등 센서 데이터, 로봇 현재 좌표, 속도 명령, 시스템 상태 정보 등.
* 서비스 (Service): 즉각적인 상태 변경 및 질의
  * LED On/Off, 모터 토크 설정, 기하학적 계산(IK/FK) 요청, 파라미터 업데이트 등.
* 액션 (Action): 장시간 소요되는 복합 태스크
  * 네비게이션(목적지 이동), 매니퓰레이터 물건 파지, 복잡한 일련의 시퀀스 제어 등.

4.2 아키텍처 설계를 위한 선택 결정 트리

시스템 설계자는 다음의 우선순위에 따라 통신 방식을 결정해야 합니다.

1. 데이터가 연속적인 스트림인가? → 토픽(Topic) 선택.
2. 일회성 명령이며 즉각적인 결과 확인이 필요한가? → 서비스(Service) 선택.
3. 작업이 완료될 때까지 시간이 오래 걸리는가?
  * 작업 중 피드백이 필요한 경우
  * 작업 중간에 **취소(Cancel/Preemption)**가 가능해야 하는 경우
  * → 반드시 액션(Action) 선택. (서비스로 긴 작업을 처리할 경우 응답 대기로 인해 시스템이 멈춘 것처럼 보일 수 있음)

5. 학습 요약 및 정리

* 핵심 차이점: 토픽은 비동기 단방향 스트림, 서비스는 동기 양방향 질의, 액션은 동기/비동기가 혼합된 복합 태스크 제어 방식입니다.
* 인터페이스 확장성: 서비스는 1개, 액션은 2개의 구분자(---)를 통해 데이터를 논리적으로 분리하며, 모든 형태는 기본 msg 정의를 확장하여 사용합니다.
* 액션의 내부 메커니즘: 액션은 완전히 독립된 기술이 아니라, 기존의 서비스와 토픽을 정교하게 조합한 형태입니다. 구체적으로는 3개의 서비스(send_goal, cancel_goal, get_result)와 2개의 토픽(feedback, status)으로 구성되어 있습니다.
* 결론: 로봇 소프트웨어 설계 시 단순히 데이터를 주고받는 것을 넘어, 동기/비동기 특성에 따른 리소스 관리와 작업 취소(Preemption) 가능 여부를 고려하여 최적의 통신 방식을 선택하는 것이 중요합니다.
